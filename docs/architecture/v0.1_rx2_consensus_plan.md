## 1. Decisions reference

These are settled by consensus. No further debate without explicit
re-evaluation captured here.

### 1.1 Channel ID convention (host-side, NOT wire-protocol DDC index)

Lyra reserves the following **host channel IDs** from v0.1 phase 0
forward. See §2.1 for the critical distinction between host channel
IDs and wire-protocol DDC indices — they are NOT the same numbering.

| Host ID | Role | Used in |
|---|---|---|
| 0 | RX1 main — **doubles as PS feedback I on HL2 during MOX+PS** (cntrl1=4 reroutes DDC0 ADC source to PA coupler) | v0.0.9.x (existing); state-product on v0.3 |
| 1 | RX1 sub-receiver | reserved (post-v0.3) |
| 2 | RX2 main — **doubles as PS feedback Q on HL2 during MOX+PS** (DDC1 sync'd to DDC0 at TX freq) | v0.1; state-product on v0.3 |
| 3 | (HL2) unused — DDC2/DDC3 are gateware-disabled in all states; (5-DDC ANAN P1) DDC3+DDC4 twist→source 1 lands here, idle when PS not armed, may carry secondary feedback during PS+TX per family (note: ANAN RX2 lives on host channel 2 via DDC2 → source 2, NOT here — corrected Round 2 Agent D B-6) | reserved |
| 4 | TX main (TXA) | v0.2 |
| 5 | PS feedback A on future ANAN 5-DDC (DDC0+DDC1 twist→source 0) | reserved |
| 6 | PS feedback B on future ANAN 5-DDC (DDC3+DDC4 twist→source 1) | reserved |

**On HL2 (4-DDC, the only Lyra v0.1-v0.3 target) — corrected by
Round 1 Agent A (2026-05-11):** PS feedback does **NOT** flow
through DDC2+DDC3 twist as earlier drafts claimed.  Thetis source
(`console.cs:8469-8488`) proves HL2 PS+TX enables ONLY DDC0+DDC1
(SyncEnable=DDC1 pairs DDC1 with DDC0 at TX freq); DDC2 and DDC3
are **disabled** by the gateware during HL2 PS+TX — those EP6
byte slots contain zeros, not feedback.  The PS feedback ADC
source is routed via `cntrl1 = 4` (frame 4, addr 0x0e) which
mux-switches DDC0's ADC input from antenna to the PA coupler.
DDC1 carries the sync-paired sample.

**Operative consequence for v0.3 PS:** the PS calcc consumer
reads from host channels **0 and 2** (which carry DDC0 and DDC1
respectively per the §4.2 wire-protocol dispatch), NOT from host
channel 3.  Host channel 3 is unused during HL2 PS+TX.  This is a
state-product reroute: outside of MOX+PS, channels 0 and 2 carry
their normal RX1/RX2 audio; inside MOX+PS, the SAME wire dispatch
delivers PS feedback I/Q to those host channels.  The DSP
consumer downstream of host-channel-0 / -2 must consult
`(mox, ps_armed)` state to decide whether to demod-and-route-to-
audio (RX path) or correlate-and-pump-to-calcc (PS path).
RX2-sub is not enabled on HL2 (Thetis convention; round-2
verified at `GetDDC` in console.cs:8579-8775).
Slots 5 and 6 stay reserved for v0.4 ANAN support.

### 1.2 Architectural building blocks

These three pieces of infrastructure must land in their respective
phases or downstream phases will require refactor:

- **AAmixer-equivalent stream router** (§3.3 in this doc): a
  2-D state matrix `(stream × destination)` gated by
  `(power, MOX, diversity, PS_enabled, RX2_enabled)`. Lyra
  currently has a single one-RX-to-stereo path. New module.
  **Indexing scheme decision (locked):** stream IDs are the
  Lyra host channel IDs from §2.2 (0 = RX1, 2 = RX2, 4 = TX-mon,
  3/5/6 for PS feedback). Lyra does NOT introduce a separate
  `mixinid()`-equivalent index space (the way Thetis does in
  cmaster.c:533, 552). Rationale: Thetis's two-axis indexing is
  a WinForms-era artifact; Lyra's host channel IDs are stable
  and unique enough to serve as both DSP channel keys AND mixer
  stream keys without collision.
- **Per-channel patchpanel block** (§5.1): the audio pan / binaural
  / I-Q-copy block at the END of each RX's DSP chain. Pan curve
  per WDSP `SetRXAPanelPan`. **NOT at the audio sink** — at the
  end of each channel's mono chain.
- **TX I/Q tap upstream of (future) iqc** (§8.2): a tap point that
  copies pre-predistorter TX I/Q into a snapshot ring. Must land
  in v0.2 even though iqc/calcc don't exist until v0.3. Otherwise
  v0.3 forces a TX-path refactor.

### 1.3 Threading model

- **v0.1, v0.2:** keep Lyra's existing single DSP worker thread
  with per-channel state. Multiplex per-channel work in the same
  worker.
- **v0.3:** switch to **per-channel worker** following Thetis's
  `_beginthread(wdspmain, ...)` pattern. PS feedback DDCs need
  parallel time-aligned snapshot capture against TX I/Q; muxing
  in a single worker introduces variable-latency between the two
  snapshot streams that calcc cannot tolerate.

### 1.4 UI deliberate divergence from Thetis

Lyra's v0.1 UI is NOT a Thetis port — these are documented
deliberate divergences:

| Lyra | Thetis | Justification |
|---|---|---|
| Hybrid focus model: single Mode+Filter / DSP+Audio panel + per-RX status badges + click-to-focus + Ctrl+1/Ctrl+2 hotkeys | Duplicated controls per RX (`comboAGC` + `comboRX2AGC`, etc.) | Smaller layout, halves disabled-control surface, cleaner in PySide6 layouts |
| Per-RX display pan + per-RX zoom | Global display pan, per-band-per-RX zoom | Asymmetry in Thetis is a WinForms artifact; PySide6 makes per-RX trivial |
| Visible AAmixer-state indicator | None — operator infers from chkPower/chkMOX/chkRX2/PS button states | UX win; Lyra adds small status badge |
| PS hardware-mod attestation checkbox | None | Safety improvement Thetis does not have |

| Shared click-tune mode across both panadapters | Independent per-pan click-tune mode (instance field on PanDisplay + two operator checkboxes) | Operator confusion cost > rare mixed-mode use case; see §6.4 |

## 2. Channel ID convention

### 2.1 Two distinct numbering axes — DO NOT CONFLATE

Lyra has TWO numbering systems that look similar but mean different
things. Confusing them is a near-certain implementation bug:

- **Wire-protocol DDC index** (0, 1, 2, 3) — refers to the four
  DDCs in the HL2's gateware, addressed via C&C frames 2 (DDC0),
  3 (DDC1), 5 (DDC2), 6 (DDC3). This is what CLAUDE.md §6.4
  refers to with `ddc[0].freq_source = VFOA`. EP6 sample slots
  carry DDC0..DDC3 I/Q in a fixed wire-format order.
- **Host-side channel ID** (0..6) — refers to Lyra's per-channel
  DSP state objects (instances of `lyra/dsp/channel.py`). Used
  internally to route audio, configure DSP per-receiver, and key
  the AAmixer routing matrix.

**The mapping between them is NOT identity, AND it is
state-product-dependent (corrected Round 1 2026-05-11):**

| Wire DDC | Host channel ID (RX-only state) | Host channel ID (HL2 MOX+PS state) | Role |
|---|---|---|---|
| DDC0 | 0 | 0 (with consumer-switched to PS calcc) | RX1 ↔ PS feedback I via cntrl1=4 |
| DDC1 | 2 | 2 (with consumer-switched to PS calcc; sync'd to DDC0 at TX freq) | RX2 ↔ PS feedback Q |
| DDC2 | 3 (via twist(2,3) dispatch — discarded as zeros when no PS-hardware variant uses these) | unused (gateware disables) | (HL2) idle; (5-DDC ANAN) RX2 main on source 2 |
| DDC3 | 3 (via twist(2,3) — discarded) | unused (gateware disables) | (HL2) idle; (5-DDC ANAN) PS feedback B on source 1 |

The EP6 dispatch in §4.2 uses a `twist()` helper to interleave
DDC2+DDC3 pairs into a single buffer for the dispatch path — but
on HL2 those slots carry zeros in all states (RX-only AND MOX+PS),
because the HL2 gateware never enables DDC2/DDC3.  HL2 PS feedback
is delivered through DDC0+DDC1 with cntrl1=4 routing the PA coupler
to DDC0's ADC input; the consumer downstream of host channels 0
and 2 must consult `(mox, ps_armed)` state to decide whether to
treat those samples as RX audio or as PS calcc input.  Plan §9.1
clarifies the full state-product reroute.

### 2.2 Locked host channel ID assignments

```
Lyra host channel ID assignments (LOCKED at v0.1 phase 0):

  0 = RX1 main                                    [active in v0.0.9.x]
  1 = RX1 sub-receiver                            [reserved, post-v0.3]
  2 = RX2 main                                    [active in v0.1]
  3 = (4-DDC HL2 only) PS feedback alias OR
      (5-DDC ANAN future) RX2 sub-receiver        [reserved]
  4 = TX main (TXA)                               [active in v0.2]
  5 = PS feedback A on 5-DDC ANAN (future)        [reserved]
  6 = PS feedback B on 5-DDC ANAN (future)        [reserved]
```

**On HL2 (4-DDC, the only Lyra-targeted hardware in v0.1-v0.3) —
corrected Round 1 (2026-05-11):** PS feedback flows through
DDC0+DDC1 (with cntrl1=4 mux-routing DDC0's ADC source from
antenna to the PA coupler, and DDC1 sync'd to DDC0 at TX freq).
Host channels 0 and 2 carry PS feedback I/Q during MOX+PS;
they revert to RX1/RX2 audio outside MOX+PS.  Host channel 3 is
unused on HL2 in all states (DDC2/DDC3 are gateware-disabled
on HL2 PS+TX per `console.cs:8469-8488`).  RX2-sub is not enabled
on HL2 (Thetis convention; round-2 verified at `GetDDC` in
console.cs:8579-8775).

**On future ANAN 5-DDC hardware (v0.4+):** PS feedback A goes to
host channel 5 via the DDC0+DDC1 twist→source 0 path; PS feedback B
goes to host channel 6 via the DDC3+DDC4 twist→source 1 path.
RX2 stays live during ANAN PS+TX via DDC2 → source 2 → host
channel 2 — this is ANAN's structural advantage over HL2.  The
correct nddc=5 dispatch (per `networkproto1.c:554-558`) is
documented in CLAUDE.md §6.7 discipline #6.

**HL2 RX2 is dropped during PS+TX (Amendment A2, Round 3
2026-05-11, narrative confirmed by Round 1 (2026-05-11) — but
the *mechanism* is different from the earlier draft):** HL2's
4-DDC silicon cannot simultaneously deliver RX1/RX2 audio AND PS
feedback.  When MOX engages with PS armed, the gateware
re-routes DDC0 (which normally carries RX1 audio) to the PA
coupler via cntrl1=4, and pairs DDC1 to it at TX freq for
quadrature sync.  RX2's true band content at VFO B is no longer
flowing — DDC1 is at TX freq, not RX2 freq.  RX2 audio fades
to silence on PS+TX onset (same ~50 ms fade as the RX-mute in
§6.6 PTT state machine).  Both panadapters follow the §9.5
source-switch matrix (RX1/RX2 panadapter both show PA-feedback
content during MOX+PS, revert to their respective RX feeds on
PTT release).  Both RX status badges display "PS-paused" (badge
distinct from operator-muted state — see §6.5 v0.3 mock).  This
is a **deliberate HL2 limitation** — ANAN 5-DDC hardware (v0.4+)
can keep RX2 live during PS+TX because DDC2 is an independent
RX2 path while DDC0/DDC1 carry PS feedback.  Lyra's PS UI must
clearly indicate "PS-paused" so operators don't think Lyra
crashed RX2.  See §9.5 for panadapter behavior and §9.3 Gate 1
bench-test confirmation.

**Round 1 error caught in round 2:** TX host channel ID was
reported as `10` (CMsubrcvr × CMrcvr = 2 × 5). Round 2 verified
Thetis source shows `2 × 2 = 4` for the standard config (CMrcvr=2,
CMsubrcvr=2 on HL2). Plan now correct.

### 3.3 Risks for phase 0

- **AAmixer stream-ID indexing**: Thetis uses `WDSP.id(rx, sub)`
  (= `mixinid` flattening 2-axis `(rx, sub)` to a single index per
  cmaster.c:533, 552), where streams become **bit positions in a
  32-bit mask** for `SetAAudioMixStates(valid_mask, active_mask)`.
  Lyra DOES use the bitmask pattern but with Lyra host channel IDs
  as bit positions.  IM-3 (Round 1 2026-05-11): the AAmixer state
  machine is **8-way** under HL2 (Power × MOX × diversity × PS,
  most cases collapsing on no-power-no-MOX-no-PS), NOT 4-way as
  earlier audio_architecture.md §2.4 implied.  Operator-mute
  toggles (MuteRX1OnVFOBTX, MuteRX2OnVFOATX) live as
  **post-mixer per-stream multipliers**, NOT state-axis additions
  (otherwise the matrix explodes 4×).  Phase 0 stubs the diversity
  axis to 0 and the TX-mon stream to silent; v0.2 activates
  TX-mon; v0.3 activates PS-disable-RX rule.
- **Patchpanel placement**: putting pan at the audio sink instead
  of in-channel will fight the AAmixer in phase 2. Round-2 DSP
  agent confirmed Thetis places this in-channel (`patchpanel.c:158-
  176`, `RXA.c:531-542`). **Lyra MUST follow this placement.**
  **IM-4 (Round 1 2026-05-11) — pick ONE pan implementation site:**
  use WDSP cffi internal patchpanel via `SetRXAPanelPan(ch, pan)`,
  NOT a Python-side reimplementation in `mix.py`.  Reason:
  running both would produce double-pan + cumulative gain math.
  WDSP cffi internal also gets the §14.10 `SetRXAPanelBinaural`
  fix interaction for free (single source of truth in C).
  Document in §5.1 that pan writes to `panel.gain2I` /
  `panel.gain2Q` (NOT `panel.gain1` which stays as per-channel
  master AF gain at WDSP's default 4.0 — although Lyra explicitly
  sets it to 1.0 since volume is applied Python-side).  L-8 pan
  curve variable naming clarification: local `gain1/gain2` from
  `SetRXAPanelPan` are stored into struct's `gain2I/gain2Q` slots,
  not back into struct's `gain1`.
- **Lambda lifetimes in Settings dialog**: the QCheckBox/QLabel
  crashes are the same pattern. Single coordinated fix needed:
  disconnect signals on dialog close, or use `weakref` lambdas.
- **M-2 SpectrumSourceMixin (REWRITTEN Round 3 2026-05-11 per
  Round 2 Agent E Gap L + Agent D B-7):** the panadapter
  source-switch contract (§9.5) is load-bearing for v0.2/v0.3 but
  the v0.0.9.9.1 EiBi-overlay precedent proves "wire it in one
  backend, miss it in another" is a Lyra-specific failure pattern.
  Earlier draft said "add to the SpectrumPanel base class" but

### 4.2 EP6 dispatch (terminology aligned Round 3 2026-05-11 per R3-6)

The 19-slot × 26-byte EP6 layout at nddc=4 must dispatch four DDC
streams. Per `networkproto1.c:383-387` (HL2 read) and `:549-553`,
the dispatch is:

```
DDC0          → xrouter source 0 → host channel 0  (RX1)
twist(DDC2,DDC3) → xrouter source 1 → host channel 3  (idle on HL2:
                                                       DDC2/DDC3
                                                       gateware-
                                                       disabled per
                                                       §2.1 CR-1)
DDC1          → xrouter source 2 → host channel 2  (RX2)
```

**Terminology consistency (Agent D B-4, fixed Round 3):** the
numbers 0/1/2 are **xrouter source IDs** (per Thetis
`networkproto1.c:263-274` `twist()` and `xrouter()` semantics),
NOT "audio chain IDs" (earlier draft used three different terms
for the same concept).  The mapping from xrouter source → host
channel uses Lyra's host-ID convention from §1.1 / §2.2.

**Phase 1 deliverables (NEW Round 3 2026-05-11 per Agent E Gap H +
synthesis R3-3):**

1. **Rewrite `lyra/protocol/stream.py::_decode_iq_samples`** to
   handle the nddc=4 26-byte slot stride per CLAUDE.md §3.3.
   Output: per-DDC IQ ndarrays `{0: ndarray, 1: ndarray, 2:
   ndarray, 3: ndarray}` with mic samples in a separate slot.
   Each ndarray has 19 complex samples per USB block × 2 blocks
   = 38 complex per UDP datagram per DDC.  The existing
   single-DDC 8-byte decode path is retired.
2. Add `twist(samples, ddc_a, ddc_b, dest_source)` helper matching
   `networkproto1.c:263-274` semantics — interleaves two DDCs
   into a 4-channel buffer (I0,Q0,I1,Q1) for the destination.
3. Add `dispatch_ddc_samples(state, decoded)` method that consults
   `Radio.dispatch_state` (per §4.2.x dispatch state contract
   below) to route per-DDC samples to the correct consumer.
   Three dispatch calls per parsed EP6 frame in the RX-only state:
   DDC0→host_ch_0, twist(2,3)→host_ch_3, DDC1→host_ch_2.

In phase 1, host channel 3 (PS feedback alias on HL2) discards
its samples (phase 3 wires the consumer).  Host channel 2 (RX2)
goes to channel 2's input.  Channel 2 has a stub DSP chain that
produces zero audio (audio routing comes in phase 2).

### 4.2.x Dispatch state contract (NEW Round 3 2026-05-11 per R3-3)

The plan's §9.5 panadapter source-switch + CLAUDE.md §14.6
captured-profile bypass + §6.7 #6 family-specific DDC mapping
all hinge on a "state-product" that drives runtime routing
decisions.  This subsection makes the contract concrete.

**`DispatchState` dataclass** (lives in `lyra/radio_state.py` or
attached to `Radio`):

```python
from dataclasses import dataclass, replace
from enum import Enum

class RadioFamily(Enum):
    HL2 = "hl2"
    HL2_PLUS = "hl2_plus"
    ANAN_P1_5DDC = "anan_p1_5ddc"
    ANAN_P2 = "anan_p2"
    # v0.4 expansion

@dataclass(frozen=True)
class DispatchState:
    mox: bool = False
    ps_armed: bool = False
    rx2_enabled: bool = False
    family: RadioFamily = RadioFamily.HL2
```

**Ownership + lifecycle:** single instance owned by `Radio`.
The only allowed mutation pattern is via `dataclasses.replace`
— field-by-field in-place assignment is forbidden (the dataclass
is `frozen=True` which enforces this at the language level).
`Radio.snapshot_dispatch_state() -> DispatchState` returns the
current reference (atomic read in CPython under the GIL).

**State updates:** Qt main thread updates `Radio.dispatch_state`
via:
- `Radio.set_mox(bool)` — on PTT/MOX/CW key edges.  Emits Qt
  signal `dispatch_state_changed(DispatchState)`.
- `Radio.set_ps_armed(bool)` — on PS dialog FSM transitions.
- `Radio.set_rx2_enabled(bool)` — on RX2 toggle.
- `Radio.set_radio_family(RadioFamily)` — on discovery
  (one-shot, at connection time; rarely changes after).

Each setter does `self._dispatch_state = replace(self._dispatch_state, mox=…)`
and emits the changed signal.

**Reader semantics:** `HL2Stream._rx_loop` reads
`state = radio.snapshot_dispatch_state()` ONCE per UDP datagram
before iterating per-DDC dispatch.  Mid-datagram MOX edges are
**coalesced to the next datagram boundary** (~1 ms granularity
at 192 kHz).  This avoids torn reads while keeping the lock-free
hot path simple.  Same pattern for `Radio._do_demod_wdsp` —
snapshot once per WDSP block.

**Dispatch table:** the per-family DDC→consumer mapping is a
pure function `radio.protocol.ddc_map(state) -> Dict[int, ConsumerID]`
with no internal state.  Consumer enum:

```python
class ConsumerID(Enum):
    RX_AUDIO_CH0 = "rx_audio_ch0"       # → host channel 0 demod chain
    RX_AUDIO_CH2 = "rx_audio_ch2"       # → host channel 2 demod chain
    PS_FEEDBACK_I = "ps_feedback_i"     # → calcc input I (HL2 DDC0 on cntrl1=4)
    PS_FEEDBACK_Q = "ps_feedback_q"     # → calcc input Q (HL2 DDC1 sync to DDC0)
    PANADAPTER_TAP_RX1 = "panadapter_rx1"
    PANADAPTER_TAP_RX2 = "panadapter_rx2"
    DISCARD = "discard"
```

Example HL2 dispatch table (state `(mox, ps_armed) → ddc_map`):

| State | DDC0 | DDC1 | DDC2 | DDC3 |
|---|---|---|---|---|
| `(False, *)` RX-only | RX_AUDIO_CH0 | RX_AUDIO_CH2 | DISCARD | DISCARD |
| `(True, False)` MOX, PS off | RX_AUDIO_CH0 | RX_AUDIO_CH2 | DISCARD | DISCARD |
| `(True, True)` MOX+PS armed | PS_FEEDBACK_I | PS_FEEDBACK_Q | DISCARD | DISCARD |

(The "MOX, PS off" row keeps DDC0/DDC1 on RX audio because the
operator's MuteRX*OnVFOBTX gates the audio at the AAmixer level
per §8.1; the protocol layer still delivers RX-band content.)

**Captured-profile bypass call site (L-9 fix, R3-3
follow-through):** `Radio._do_demod_wdsp` reads the state once
per WDSP block:

```python
def _do_demod_wdsp(self, iq, frame_stats):
    state = self.snapshot_dispatch_state()
    bypass_capture = state.mox and state.ps_armed

    if self._iq_capture_profile is not None and not bypass_capture:
        iq = self._iq_capture_profile.apply(iq)
    elif bypass_capture and self._captured_profile_was_active:
        # falling edge of (was-not-bypassed, now-bypassed)
        self._iq_capture_profile.reset()
    self._captured_profile_was_active = not bypass_capture
    # ... hand iq to _wdsp_rx.process(iq)
```

On the rising edge of MOX+PS (entering bypass): pre-pass is
skipped this block.  On the falling edge (returning from
MOX+PS to either RX-only or MOX-no-PS): `reset()` flushes the
stale STFT overlap-buffer so the resumed pre-pass starts clean.
Operator may see a brief (~200 ms) noise-floor adjustment as
the rolling buffer repopulates — surface as a faint "RX2 resuming"
badge if operator-visible UX demands it (deferred to v0.3 UX
polish).

**Phase 1 verification (NEW step 7):** programmatically toggle
`radio.dispatch_state.mox` via test injection (no real PTT) and
verify `radio.protocol.ddc_map(state)` returns the expected
per-DDC consumer routing.  Tests the abstraction even though
v0.1 has no real PS consumers wired yet.  Catches Phase 1
state-product abstraction bugs before v0.2 wires real consumers.

**Threading model summary:**
- Qt main thread: writes (one writer).
- `HL2Stream._rx_loop` thread: reads (snapshot per datagram).
- `DspWorker` thread: reads (snapshot per WDSP block).
- CPython GIL ensures the dataclass reference read is atomic.
- No locks needed.  No `threading.Lock` anywhere in the hot path.

### 4.3 Risks for phase 1

- **Byte stride math**: at nddc=4, EP6 sample-set stride is
  `6 × nddc + 2 = 26 bytes` per slot, 19 slots per USB block. Mic
  is at offset `8 + nddc × 6` per slot. Off-by-one here is the
  most common first-try failure mode. Round-2 protocol agent
  identified this explicitly.
- **PTT-edge handling**: at nddc=4, the `out_control_idx = 2` jump
  that nddc=2 mode uses (`networkproto1.c:886-891`) is **skipped**.
  This is correct because DDC0 stays on RX1 freq at nddc=4 — but
  Lyra's existing code may have inherited the nddc=2 jump logic;
  needs audit.
- **`I2C inline branch`**: C0 bit 7 of returned C&C bytes signals
  I2C readback data, NOT regular telemetry. Parser must check
  bit 7 BEFORE dispatching on `C0 & 0xf8` for telemetry sub-cases
  (`networkproto1.c:478-493`). Easy to miss; corrupts FwdPower /
  RevPower / supply / temp readouts if missed.
- **Mic decimation counter persistence**: HL2 mic is in EP6 at
  a different rate than I/Q; decimation counter must persist across
  EP6 frames (`networkproto1.c:278, 400-410, 566-576`).
- **Watchdog**: 3-second EP6 silence triggers `destroy_pro` (line
  443). RX2 work doesn't change this risk; band changes during
  phase 1 testing should not stall EP6 reads >3s.

6. **Priming-frame duplex bit verification (Amendment A5,
   originally Round 3 2026-05-11, corrected by Round 1 Round-1
   2026-05-11 agent A).** Critical HL2 quirk: per CLAUDE.md §3.2,
   the duplex bit (C4 bit 2) is required on every **main-loop**
   frame-0 emission or VFO freq updates silently fail to apply.
   Subtlety the earlier draft of A5 missed: **the priming frames
   themselves do NOT include the duplex bit** — `ForceCandCFrames`
   at networkproto1.c:111-127 emits priming with the hard-coded
   layout `C4 = (nddc-1) << 3 = 0x18` (no duplex bit), and the
   gateware accepts the priming VFO writes regardless.  The

   datagrams (= 12 USB sub-frames; only DDC0/RX1 is primed —
   DDC1/RX2 initial state is gateware-default until the first
   main-loop case-3 emission, per agent A finding), and (b) the
   first ~10 main-loop frames after priming.

   **Pass criteria:**
   (a) priming USB sub-frame 1 (general-settings) has `C0 = 0`
       and `C4 = 0x18` (NOT 0x1C — no duplex bit in priming).
   (b) priming USB sub-frame 2 (VFO-write) has `C0 = 2` (TX VFO)
       or `C0 = 4` (RX1 VFO) and C1..C4 = freq bytes.
   (c) FIRST main-loop emission after priming: frame-0 C4 = 0x1C
       (duplex bit present, nddc=4 confirmed via bits[6:3]=0x18).
   (d) RX1 audio appears within ~140 ms of priming completion
       on a known carrier (e.g., WWV 10 MHz).
   (e) Tuning VFO A (RX1) AFTER priming moves DDC0's effective
       frequency — verifies the duplex bit is actually enabling
       VFO updates.
   (f) Tuning VFO B (RX2) AFTER first main-loop case-3 (~17
       round-robin cycles in, ~7 ms at full rate) moves DDC1's
       frequency.

   **Fail modes:**
   - Priming C4 = 0x1C (should be 0x18): means Lyra is over-
     applying the duplex bit, likely also breaking gateware-
     specific assumptions about clean priming state.
   - Main-loop C4 = 0x18 (should be 0x1C): missing duplex bit

     failure mode for HL2 multi-DDC enablement and produces a
     maddening "tuning seems to work but the radio doesn't
     follow" symptom.  Gate guarantees we catch this at phase 1
     not phase 3.

## 5. Phase 2 — RX2 audio routing (v0.1.0-pre3)

**Goal:** RX2 produces audio. Default routing: RX1 hard-left,
RX2 hard-right.

### 5.1 Per-channel patchpanel block

Each RX channel gets its own patchpanel block at the end of its
DSP chain (after all per-RX DSP: notches, demod, NR, AGC, APF, etc.).
The block:

- Takes mono float32 audio (channel's output).
- Produces (I, Q) lane pair per the WDSP pan curve from
  `patchpanel.c:55-101, 158-176`:
  - `pan ≤ 0.5`: `gain1 = 1, gain2 = sin(pan × π)`
  - `pan > 0.5`: `gain1 = sin(pan × π), gain2 = 1`
  - `output_I = audio × gain1, output_Q = audio × gain2`
- Has independent operator-controllable `pan` per channel.
- Default for v0.1: RX1 `pan = 0` (hard-left, all in I lane);
  RX2 `pan = 1` (hard-right, all in Q lane).

The patchpanel block also handles binaural and I-Q-copy modes
(future, v0.1.0+ polish).

### 5.2 AAmixer stream router

Activates more routes in phase 2:

| Stream | Default destination |
|---|---|
| Channel 0 (RX1) `pan` output | I lane of stereo audio sink |
| Channel 2 (RX2) `pan` output | Q lane of stereo audio sink |

The mixer state matrix is gated by:

- `power` (off → all streams muted)
- `MOX` (TX active → optionally mute one or both RX per
  `MuteRX1OnVFOBTX` / `MuteRX2OnVFOATX` operator toggles)
- `diversity` (when on, RX2 routing is reconfigured but the
  operator's RX2-enable state is preserved per round-2 UI agent
  finding)
- `PS_enabled` (during PS+MOX, both RX audio routings are disabled
  per console.cs:28307)
- `RX2_enabled` (off → channel 2 stream output muted but channel
  still runs)

### 5.3 Audio chain integration

```
[Channel 0]  WDSP RXA cffi engine (single fexchange0 call):
               decim → notches (NBP0) → NR (EMNR) → ANF → LMS (ANR)
                → AGC (wcpagc) → APF (xspeak biquad)
                → patchpanel (xpanel via SetRXAPanelPan, IM-4)
                → resample-out
             then Python-side post-WDSP:
                → Volume + Mute → BIN (optional)
                → AAmixer route host_ch_0 → L

[Channel 2]  WDSP RXA cffi engine (single fexchange0 call):
               decim → notches → NR → ANF → LMS → AGC → APF
                → patchpanel → resample-out
             then Python-side post-WDSP:
                → Volume + Mute → BIN (optional)
                → AAmixer route host_ch_2 → R
                                              → sink
```

**Chain-ordering note (CORRECTED Round 3 2026-05-11 per Round 2
Agent D Cross-file §C):** earlier draft showed APF AFTER
patchpanel, which contradicts WDSP-internal order.  Verified
against CLAUDE.md §14.2 + RXA.c: WDSP runs `xspeak → xmpeak →
xssql → xpanel` — APF (= xspeak biquad) sits BEFORE the
patchpanel inside the cffi engine.  The diagram above now
reflects WDSP-internal order: APF → patchpanel → resample-out
all inside one `_wdsp_rx.process(iq)` call.  Python-side
post-WDSP work (Volume, Mute, BIN, AAmixer route) happens AFTER
the cffi engine returns.

**Note (Round 1 2026-05-11 cleanup, §15.2 RX2-plan leveler refs):**
the Audio Leveler module was DELETED in v0.0.9.6 cleanup arc
(CLAUDE.md §14.9).  Earlier drafts of this section showed
`→ leveler` after Volume — that's stale.  WDSP AGC subsumes the
dynamic-range function leveler used to provide.  The chain above
reflects the post-v0.0.9.6 reality.

Note: each channel has its OWN AGC and APF instance. Each channel
independently configurable (different mode, different filter,
different AGC profile per the focused-RX UI model in phase 3).

### 5.4 Risks for phase 2

- **Pan placement**: must be per-channel, before AAmixer. Round-2

## 6. Phase 3 — RX2 UI integration (v0.1.0-pre4)

**Goal:** Operator-facing controls for RX2.

### 6.1 Hybrid focus model

(Re-stated from §1.4.) Single Mode+Filter and DSP+Audio panel
operate on the focused RX. Click any RX freq display to focus.
Hotkeys Ctrl+1 → focus RX1, Ctrl+2 → focus RX2. Per-RX status
badges (mode, filter abbreviation, AGC profile) display read-only
on each freq display.

This is a **deliberate divergence** from Thetis's duplicated-
controls model (verified by round 2 UI agent finding 35
`comboRX2*`/`chkRX2*` instances).

**Focus-vs-MOX rule (M-1, Round 1 2026-05-11):** focus is
operator-controlled.  MOX never auto-shifts focus.  MOX never
blocks focus changes.  The Mode+Filter+DSP panel is **enabled iff
the focused RX is not the current TX VFO target** — computed from
`tx_freq()` (§6.2): if `focused_rx == TX-VFO-RX` then panel
disables the `MHz` spinbox, the mode picker, and the filter
picker (the things that would silently divert the carrier
mid-TX).  AGC, NR, ANF, squelch, AF gain stay **enabled** —
those are RX-side knobs and shouldn't be locked by TX state.

**Focus-vs-PS-paused-RX rule (M-1, Round 1 2026-05-11):** during
HL2 PS+TX with RX2 paused (per §2.2 CR-1 correction), if operator
focuses RX2, the panel shows the operator's current RX2 settings
with a non-modal banner: "RX2 paused for PureSignal — changes
apply on PTT release."  Changes queue and apply on MOX release.
Do **NOT** lock the panel; that would be hostile to operators
who want to prep RX2 settings for after the QSO.

Lyra implements `tx_freq()` as a pure function of state per the
TXFreq derivation table at `console.cs:11396-11420`:

```python
def tx_freq(rx2_enabled: bool, vfoatx: bool, vfobtx: bool,
            split: bool, vfoa: int, vfob: int, vfoasub: int) -> int:
    if not rx2_enabled:
        if vfobtx:
            return vfob

Invariants:

- VFOATX and VFOBTX mutually exclusive (radio-style).
- SPLIT + RX2 + VFOBTX requested → force VFOATX = true
  (per console.cs:36540).
- SPLIT off without RX2 → VFOSync forced off.
- During MOX, VFO setters early-return when `MOX && (VFOATX ||
  (!RX2_enabled && VFOBTX))`.
- **M-9 Lock B semantics (Round 1 2026-05-11):** Lock B means
  "don't change the FREQUENCY of VFO B."  Operations that change
  VFO B's freq (manual tuning, Swap, A→B, B→A) are **blocked**
  while Lock B is on.  Operations that don't change VFO B's freq
  (SPLIT toggle, VFOATX toggle, RX2 enable/disable) are NOT
  blocked.  Operator gets a small toast on blocked attempt:
  "VFO B is locked — unlock first."

Buttons: A→B, B→A, Swap (full state when RX2 enabled, freq+mode+

### 6.4 Click-tune divergence (corrected Round 1 2026-05-11)

**Earlier draft's reasoning was based on a factually wrong Thetis
claim** — Round 1 Agent C verified at `PanDisplay.cs:478` that
the relevant field is `private ClickTuneMode current_click_tune_mode`
(an **instance field**, not static).  Console.cs:10894 / 10906
also confirm two **separate per-pan flags** (`_click_tune_display`
for RX1, `_click_tune_rx2_display` for RX2) each driven by a

divergences), not §1.5 (Thetis-adopted patterns).  Lyra chooses
to **share** click-tune mode across pans because the operator-
confusion-from-divergence cost outweighs the rare mixed-mode use
case.  Hit-test routing (which pan was clicked → which RX gets
tuned) remains per-pan via independent rectangles, but the click-
tune behavior knobs (snap radius, peak-detect threshold,
modifier-key mapping, filter selection during click) are shared
state.

Implementation: ONE shared "click-tune mode" state object
owned by `MainWindow`, exposed via `radio.click_tune_settings`;
TWO separate hit-test rectangles + per-pan click handlers that
each consume the shared mode and dispatch the tune action to the
correct RX channel.  Future Settings → Spectrum option may
re-introduce Thetis-style per-pan independence if operator demand
emerges, but default behavior in v0.1-v0.3 is shared.

### 7.3 RX-side audio chain extensions slated for v0.2

The full RX audio chain in v0.1.0 reads (corrected Round 1
2026-05-11 to remove deleted-leveler ref):

```
(WDSP RXA cffi) decim → notches → NR → ANF → LMS → AGC → APF
                → Volume + Mute (Python post-WDSP) → BIN (optional)
                → sink

The legacy `leveler` block was deleted in v0.0.9.6 cleanup arc
(WDSP AGC subsumes its function).

### 7.x SettingsDialog connection-tracking refactor (parked from v0.0.9.6)

**Background.** v0.0.9.6 hit operator-visible "C++ object already
deleted" libshiboken errors when Brent (tester) cycled through
the Settings dialog several times during EiBi troubleshooting.
Mechanism: every Settings tab does many
`radio.<signal>.connect(lambda v: self.<widget>.setX(v) ...)`
calls in `__init__`, but the connections are never disconnected
on dialog close.  Closing destroys the C++ widget; the Python
closure + slot record live forever on the long-lived `Radio`
side; subsequent signal fires call the lambda which touches the
dead widget and raises.

**v0.0.9.6.x sweep — silenced the noise but not the leak.**

* Helpers added to `lyra/ui/settings_dialog.py`:
    * `_is_widget_valid(widget)` — `shiboken6.isValid` wrapper
    * `_safe_mirror(widget, getter, setter, value)` — drop-in
      replacement for the inline mirror lambda; silently no-ops
      on dead widgets
    * `@_swallow_dead_widget` — decorator for `_on_*` / `_sync_*`
      slot methods; catches the libshiboken RuntimeError, re-
      raises any other RuntimeError so genuine bugs still surface
* 10 inline mirror lambdas converted to `_safe_mirror`
* 24 slot methods (those connected to long-lived Radio signals)
  decorated with `@_swallow_dead_widget`

This eliminates the user-visible crash error class and the
console traceback noise.  It does NOT disconnect anything —
each Settings open still adds ~50–80 permanent slot
registrations to Radio's signal dispatchers that the silenced
no-ops keep walking on every signal fire.

**The actual leak — bounded but real.**

| Metric | Per Settings open-close |
|---|---|
| New permanent slots wired | ~50–80 |
| Memory cost | ~100 KB (closures + Qt slot records) |
| Dispatch cost | ~µs per signal fire (negligible) |

Realistic operator usage: 5–10 open-cycles per session, end
session with restart → ~500 KB to 1 MB leaked per session.
Lyra's already a 200+ MB resident process.  Invisible at human
time scales.  Zero user impact.

**Why we parked it for v0.1 instead of fixing in v0.0.9.6.x:**

The proper fix needs a connection-tracking discipline applied at
~80 `radio.X.connect(...)` call sites across the file — too big
to bundle quietly in a patch release alongside the operator-
visible Brent fixes.  By v0.1 the SettingsDialog is growing
again (RX2 panels, TX panels, focus-model controls), so adding
the discipline is more valuable in that release window.

**Implementation sketch (when v0.1 polish lands):**

Two equivalent approaches — pick one based on what's cleaner
once the v0.1 refactor is in flight:

*Option A — connection-tracking helper:*

```python
# In each SettingsTab base/mixin
def _track(self, signal, slot):
    signal.connect(slot)
    self._tracked_connections.append((signal, slot))

def _disconnect_all(self):
    for signal, slot in self._tracked_connections:
        try:
            signal.disconnect(slot)
        except (RuntimeError, TypeError):
            pass
    self._tracked_connections = []
```

Then sweep every `radio.X.connect(...)` to
`self._track(radio.X, ...)`.  Override
`SettingsDialog.closeEvent` to walk tabs and call
`_disconnect_all()` on each.  Or use the tab's `destroyed`
signal to fire the disconnect automatically.

*Option B — Qt context-object pattern:*

```python
# Before
radio.X.connect(lambda v: self.widget.setX(v))
# After
radio.X.connect(self, lambda v: self.widget.setX(v))
```

The 2-arg `connect(receiver, slot)` form ties the connection
lifetime to `receiver` — when `self` is destroyed, Qt
automatically disconnects.  Verify exact PySide6 syntax for
lambdas with explicit context (the form may require a
`@Slot()` decoration or a method reference rather than a raw
lambda — check at impl time).

**Why both options touch the same ~80 sites:**

The leak is per-connection; fixing one means converting every
`connect(...)` call.  The decorator+helper pair shipping in
v0.0.9.6.x stays in the file regardless — they're the
"belt-and-suspenders" defense against any new mirror site
introduced later that someone forgets to track.

**Pre-impl tasks when v0.1 picks this up:**

1. Confirm exact PySide6 syntax for the chosen approach (likely
   Option A — explicit tracking is harder to get wrong than
   relying on Qt's lifetime semantics across language
   boundaries).
2. Decide whether SettingsDialog.closeEvent or
   `tab.destroyed` is the right disconnect trigger.  Probably
   `closeEvent` since tabs may not emit `destroyed` until
   after all signals have already fired against them.
3. Test by opening Settings 100× in a row and confirming
   `len(radio.X._connections)` (or equivalent introspection)
   stays bounded.

  with AVP3-only OC pin register.  When v0.4 splits the protocol
  module per CLAUDE.md §6.7, frames 17/18 stay in `p1_hl2.py`;
  they MUST NOT be wired into a generic `p1.py` base.

### 8.2 TX I/Q tap (mandatory for v0.3)

Round-2 DSP agent flagged this as a "future-proofing" requirement
that's cheap to add now but expensive to retrofit. Phase 8.2:

- Add a tap point in `tx_channel.py` between the `bandpass` block
  and the (placeholder) `iqc` block.
- Tap copies post-bandpass / pre-iqc TX I/Q samples into a ring
  buffer.
- In v0.2, the ring buffer is allocated and filled but never read.
- In v0.3, calcc reads from this ring buffer to align with feedback
  samples.

Why this must be in v0.2: refactoring the TX path in v0.3 to add a
tap means re-validating every TX sub-mode (SSB/CW/AM/FM) for
behavior change. Adding the tap as a no-op in v0.2 means v0.3 can
focus purely on PS work.

### 8.3 v0.2 protocol items

Items round 2 protocol agent identified that are HL2-specific and
NOT in earlier Lyra docs:

- **Frame 17** (`tx_latency` + `ptt_hang`, addr 0x17,
  networkproto1.c:1162-1168). C3 = `ptt_hang & 0x1F` (5-bit, 0-31).
  C4 = `tx_latency & 0x7F` (7-bit, 0-127, ms units). Per-rig
  set-and-forget; expose in Settings → TX. Defaults TBD (verify
  against Spark/Quisk; Thetis source doesn't show defaults).
- **Frame 18** (`reset_on_disconnect`, addr 0x3a,
  networkproto1.c:1170-1176). HL2-only graceful shutdown register.
- **CWX PTT bit 3 in I-LSB** (networkproto1.c:1247-1259). On HL2,
  TX I-sample LSB carries CW state bits during CW: bit 3 = cwx_ptt,
  bit 2 = dot, bit 1 = dash, bit 0 = cwx. Non-HL2 has only 3 bits.
- **EER mode overwrites LR audio buffer** during TX
  (networkproto1.c:1224-1226). EER + TX clobbers the audio path.
- **Duplex always on**: `C4 |= 0x04` regardless of XmitBit. EP2
  always streams full-rate; XmitBit (C0 bit 0) controls PA
  keying, not audio framing.

### 8.4 TX multi-meter activation

When v0.2 SSB ships, the meter system gets its TX-side data feed
populated.  Three things land together:

**(a) Forward power + SWR signals.**  HL2 already sends raw
`fwd_pwr_adc` and `rev_pwr_adc` in EP6 telemetry frame 0x10.
Lyra's `FrameStats.fwd_pwr_adc` / `.rev_pwr_adc` parse these as
raw ADC counts (live since v0.0.9.x).  v0.2 adds:

- **Operator-tunable calibration constants** for `fwd_w =
  cal_a_fwd × fwd_pwr_adc² + cal_b_fwd × fwd_pwr_adc + cal_c_fwd`
  (quadratic fit; cal vary per HL2 board revision per CLAUDE.md
  §3.7).  Self-cal workflow lives in Settings → TX → Calibrate.
  **M-11 Round 1 2026-05-11:** quadratic fit requires **3 data
  points**.  Self-cal workflow walks operator through three
  readings: "Transmit at 10% drive → enter measured W → next →
  Transmit at 50% → enter → next → Transmit at 100% → enter →
  done."  Three points solve the quadratic.  Per-band cal
  storage (PA non-linearity is band-dependent — loading varies,
  low-pass filter responses vary).  Default cal for new HL2
  install ships a quadratic derived from published HL2-board cal
  points with an "uncalibrated — operator self-cal recommended"
  badge until the operator runs the 3-point workflow.
  Family-aware: v0.4 ANAN has different ADC scaling; cal
  constants live in `RadioCapabilities` per-family-defaults with
  operator-override.
- **`Radio.tx_pwr_w_changed(float)` signal**: emits computed
  watts when telemetry updates.
- **`Radio.tx_swr_changed(float)` signal**: emits SWR computed
  from forward + reverse:
  `rho = sqrt(rev_w / max(fwd_w, eps)); swr = (1 + rho) / max(1 - rho, eps)`.
- **Wiring** (`panels.py`): connect both signals to LitArc and
  LED meter setters, same pattern as `smeter_level` /
  `agc_action_db`.

**(b) Mic level + ALC + Compression signals.**  TX path
provides:

- `Radio.tx_mic_db_changed(float)`: pre-leveler mic input peak
  in dB.  Source: TX channel input gain stage.
- `Radio.tx_alc_db_changed(float)`: ALC headroom remaining
  (positive = room to drive louder, negative = clipping).
  Source: TX channel ALC block (`SetTXAALCSt` equivalent).
- `Radio.tx_comp_db_changed(float)`: speech compressor gain
  reduction, dB.  Lands with the WDSP `compress.c` port in
  v0.2.1 (CLAUDE.md §4.1 already targeted this).

**(c) Auto-style-switch on MOX.**  When MOX engages, meter
auto-switches to **LED bar style** (5 stacked rows visible at
once).  When MOX releases, auto-switches back to operator's
RX-mode preference (default Lit-Arc).  This matches modern
rig UX (FT-991, IC-7300, ANAN G2) and gives operators
glance-readable TX-side telemetry without chip-clicking.

**Operator override:** Settings → Visuals → Meter behavior:
- "Auto-switch to LED on TX" (default ON)
- "Stay in current style during TX" (operator picks style for
  both RX and TX)

**LED bar TX layout** (rows top → bottom; matches Lyra's
existing `LedBarMeter` scaffolding):

| Row | Live data source | Scale |
|---|---|---|
| **PWR** | `tx_pwr_w_changed` | 0..200 W |
| **SWR** | `tx_swr_changed` | 1.0:1 .. ∞ (non-linear, red past 2:1) |
| **MIC** | `tx_mic_db_changed` | -60..+5 dB |
| **COMP** | `tx_comp_db_changed` | 0..30 dB compression |
| **ALC** | `tx_alc_db_changed` | -10..+3 dB headroom |

The S row is dimmed (no audible RX during TX in normal config).
AGC row is also dimmed (RX-only metric).

**Lit-Arc TX mode chips** (for operators who prefer the big
single-arc readout): `AVAILABLE_MODES_TX = (PWR, SWR, MIC,
COMP, ALC)`.  Default chip on TX entry: PWR.

**Two-position preemph note (R3-1 / IM-5 follow-through):**
preemph appears TWICE in `xtxa()` — at line 568 (position 0,
BEFORE leveler) and line 581 (position 1, AFTER bp2).  Only ONE
of them runs at a time, gated by `SetTXAFMEmphPosition(ch, pos)`.
**FM convention is position 1** (pre-emph BEFORE deviation
modulator).  Position 0 is used in some non-FM configs.  Lyra's
mode-change handler must call `SetTXAFMEmphPosition(1)` on FM
mode entry; non-FM modes leave the default position.  Get this
wrong and FM pre-emphasis applies at the wrong stage in the
chain → wrong audio shaping on the air.

**TXA setter audit checklist (IM-5, Round 1 2026-05-11)** — to
prevent the §14.10 right-channel-silent-class bug for TX, every
`SetTXA*` call Thetis makes during channel init MUST be replicated
in Lyra's TX channel open.  Highest-risk-missed setters per
Round 1 Agent B:

1. `SetTXAPhRotRun(ch, 1)` for SSB — default in `create_txa` is
   OFF; Lyra MUST explicitly enable for SSB modes.  Skipping
   this drops PEP-PAR ~3-4 dB worse than industry baseline.
2. `SetTXABandpassFreqs(ch, low, high)` — sign convention.  USB
   uses positive freqs, LSB uses negative (WDSP mirrored-baseband,
   same gotcha as §14.2 RX side).  Get this wrong → wrong sideband
   transmitted.
3. `SetTXACompressorRun` ↔ bp1 pairing.  WDSP toggles bp1
   internally when compressor toggles; Lyra cffi port must
   replicate.  Otherwise comp-on without bp1-on produces aliased
   garbage past the SSB passband.
4. `SetTXAALCAttack / Decay / Thresh` — defaults aren't set by
   `create_txa` itself; Lyra must push.  Without explicit calls
   ALC kicks in weirdly.
5. `SetTXAosctrlRun` for CESSB — entirely missing from earlier
   plan draft; required when compressor is on or the operator
   gets splatter.
6. `SetTXAFMEmphPosition` mode-dependent — position 1 for FM
   (pre-emph BEFORE deviation modulator).

**Verification** — v0.2.0 mode-cycle bench test: transmit a tone
in USB / LSB / AM / FM and verify the EP2 TX I/Q stream produces
the expected mono baseband for SSB and expected sin/cos pair for
AM/FM.  Catches sign-flip and modulator-state bugs before flight.

## 9. v0.3 PureSignal

**Goal:** Pre-distortion calibration + correction for HL2 with the
PA-tap hardware mod installed.

### 9.1 v0.3 architecture summary

- **calcc.c port** (`lyra/dsp/ps_calcc.py`, ~1164 LOC C → ~600 LOC
  Python with NumPy). Consumes time-aligned TX-pre-iqc and feedback-
  post-DDC snapshot buffers; produces predistortion coefficient set.
- **iqc.c port** (`lyra/dsp/ps_iqc.py`, ~315 LOC C). Applies
  coefficient set to TX I/Q stream. 5-state lifecycle: RUN, BEGIN,
  SWAP, END, DONE.
- **xbuilder port** (cubic-spline coefficient builder from
  `lmath.c`).
- **delay.c port** (delay line for sample-time alignment).

### 9.2 v0.3 operator flow

- Settings → PS is a **gate** (self-attestation checkbox per
  CLAUDE.md §6.5 + "Open PS Dialog" button), NOT the calibration
  UI itself.
- **PS dialog (M-5 Round 1 2026-05-11):** non-modal `QDialog`
  (modeled on Thetis's PSForm which is a `Form`, not a modal).
  Operator can keep the dialog open alongside the main window and
  continue tuning RX1, swapping VFOs, adjusting mic gain while PS
  is active.  **Closing the dialog does NOT stop PS** — only the
  OFF state in the 8-state command FSM does.  The Tuning panel PS

### 9.3 Mandatory bench-test gates

**Round 2 protocol agent confidence for v0.3 is 60%** — the
following three items MUST be tested on N8SDR's HL2+ hardware
before declaring v0.3 ready. Without these tests, first-try
execution is not realistic:

#### Gate 1: PS hardware-mod feedback source switch (corrected Round 1 2026-05-11)

The `cntrl1` bit that switches HL2's feedback ADC from antenna
sense to TX coupler sense is **NOT a direct register**. It propagates
via:

- Console.cs:8486 sets `rx_adc_ctrl1 |= 0x04` (HL2 nddc=4 path).
  (5-DDC ANAN uses `(rx_adc_ctrl1 & 0xf3) | 0x08`, NOT "0x08 instead";
  per Agent A correction the operation is AND-mask-then-OR, not
  simple replace.  Not relevant for HL2.)
- This combined value (`P1_adc_cntrl`) is split across **two C&C
  bytes** in frame 4 (addr 0x0e):
  - **C1 = `P1_adc_cntrl & 0xFF`** — lower 8 bits (`cntrl1`)
  - **C2 = `(P1_adc_cntrl >> 8) & 0xFF`** — upper 8 bits (`cntrl2`).
    IM-2 (Round 1 2026-05-11): the mask is **0xFF, not 0x3F**.
    Thetis source (`networkproto1.c:1018`) uses
    `0b0011111111` which is 0xFF, not 0x3F.  Earlier draft's claim
    of "upper 2 bits reserved" was a hallucination.
- **TX step attenuator location:** frame-4 **C3** carries
  `tx_step_attn` at a 5-bit mask; the HL2-extended -28..+31 dB
  attenuator range lives in **frame-11 C4** (6-bit + sign per
  networkproto1.c:1099-1102, HL2 main loop case 11).  Earlier
  draft incorrectly placed TX step attn in frame-4 C2.
- Both bytes (C1 and C2) must be written every frame-4 emission.

**Bench test (corrected Round 1):** With PS armed (cntrl1=4) and
no signal, sweep TX from low to high power into a dummy load
while monitoring **DDC0+DDC1** sample magnitudes via a diagnostic.
(Earlier draft incorrectly said DDC2/DDC3 — those are
gateware-disabled during HL2 PS+TX per CR-1 correction.)
**Pass criteria:**
- DDC0+DDC1 sample magnitudes track TX power level (>20 dB swing
  across a 30 dB TX power sweep) → `cntrl1` mux routing worked.
- DDC2/DDC3 sample magnitudes flat at zero → confirms gateware
  disables DDC2/DDC3 during HL2 PS+TX.
- DDC0+DDC1 magnitudes flat (within 6 dB across the sweep) →
  `cntrl1` did not propagate; check both C1 and C2 byte writes
  in frame 4 AND confirm the PS hardware mod is actually wired
  (operator self-attestation per §1.4).

#### Gate 2: PS rate disconnect

Two rates exist:

- **Gateware DDC rate**: `rx[0].sampling_rate` for HL2 (per Lyra
  CLAUDE.md §3.8; verified by round-2 protocol agent).
- **calcc input rate hint**: 192000 (constant from
  `cmaster.cs:316/424-437`).

These are NOT the same number on HL2 unless RX1 happens to be at
192k. Lyra's `ps_calcc.py` MUST be told the actual gateware rate,
not 192000. **Or** the host must resample feedback to 192k before
calcc.

**Bench test:** Run PS calibrate at RX1 = 96k. **Pass criteria:**
- Coefficient magnitudes finite (no NaN, no Inf).
- Coefficient magnitudes within 0.1× to 10× the expected gain
  envelope for the operator's TX power level — anything outside
  this range indicates calcc consumed garbage feedback.
- Post-correction residual at the spectrum analyzer:
  3rd-order IMD products at least **−30 dBc** below the two-tone
  carriers (typical untreated HL2 sits at −20 to −25 dBc; PS
  should improve this by 10+ dB).
- Auto-attenuator FSM completes one full Monitor → SetNewValues
  → RestoreOperation cycle within 5 seconds.

If any of these fail at RX1=96k but pass at RX1=192k, the
rate-hint is wrong; calcc is being told 192000 but the gateware
delivered 96000. Fix by passing actual rx1_rate to calcc.

#### Gate 3: HL2 PS feedback complex-pair correctness (re-framed Round 1 2026-05-11)

Earlier draft of Gate 3 tested DDC2+DDC3 `twist()` interleave on
HL2 — but that's moot since DDC2/DDC3 are gateware-disabled on
HL2 PS+TX (CR-1).  HL2 PS feedback comes from DDC0+DDC1
(sync-paired with cntrl1=4 mux).  The relevant bench test is
whether DDC0/DDC1 form a coherent complex pair when the PS hardware
mod is wired and cntrl1=4 is set.

**Bench test:** With a known CW carrier on the dummy load and
PS armed (cntrl1=4), capture EP6 frames during a short MOX burst
and verify:
- DDC0 I/Q pair shows the carrier at the expected baseband
  frequency (TX freq minus DDS LO offset).
- DDC1 I/Q pair shows the SAME carrier at the SAME baseband
  frequency, sync'd to DDC0 in phase.
- Cross-correlation between DDC0 and DDC1 complex streams shows
  high coherence (>0.95) — confirms the sync-pair is delivering
  redundant samples for averaging, NOT independent samples that
  would need separate handling.

If DDC0 and DDC1 are phase-inconsistent or show different
baseband frequencies, the gateware sync-pair didn't engage —
likely cause: cntrl1 not making it to the gateware, OR the PS
hardware mod isn't actually wired to the DDC0 secondary ADC
input.

**For ANAN P1 5-DDC (future v0.4):** the corresponding gate
tests DDC0+DDC1 twist coherence via `twist(spr, 0, 1, 0)` and
also DDC3+DDC4 twist for the secondary feedback path.  ANAN P1
has TWO feedback DDC pairs; HL2 has one.

#### Gate 4: speech-content convergence (NEW Round 1 2026-05-11 per IM-6)

The 3 gates above verify the PROTOCOL plumbing.  Gate 4 verifies
the calcc MATH actually produces stable coefficients on operator-
realistic content.

**Bench test:** with PS armed and operator-grade dummy load,
transmit 30 seconds of representative speech.  Monitor calcc
state via diagnostics:
- `binfo[]` reports all xbuilder success codes (= 0).
- `convex == 1` (line 460 of calcc.c) — the calibration produced
  a monotonically-increasing gain function, not a concave-then-
  convex one that would indicate a misbehaving PA region.
- Coefficient magnitudes stable across the 30 seconds (no large
  swings that would indicate divergence).

**Fail mode:** if calcc reports `convex == 0`, the PA is going
into compression at some intermediate level and recovering at
higher level — physically unusual but possible with certain
Class-D Class-AB transition behavior.  Operator-facing message:
"PS calibration unstable; check PA drive level and operating
class."  Thetis refuses to apply non-convex corrections; Lyra
must do the same.

#### Gate 5: CW workflow guard (NEW Round 1 2026-05-11 per IM-6)

**Calcc's `xbuilder` needs a DYNAMIC envelope range** to populate
all `dog.cpi[]` bins.  For CW (constant carrier), the envelope is
flat = single x value = xbuilder underfills the bins =
`SetTXAiqcSwap` never fires = PS never updates.  This means **CW
transmits do NOT trigger PS recalibration.**  Thetis operators
know this; they two-tone-tune-up before going on the air.

**Bench test:** key CW into dummy load with PS armed for 60
seconds.  **Pass:** no `Sem_CalcCorr` signaling (dog.count
never increments), iqc state stays in current cset, no spurious
PS-yellow/red transitions on the lblPSInfoFB-equivalent.

**Operator UX requirement:** Settings → PS first-time-use checklist
must include "two-tone or VFO sweep tune-up required before
relying on PS for SSB content; CW does NOT retrigger PS calibrate."
A toast on first MOX+PS-armed+CW state would catch operators who
miss the checklist.

#### Gate 6: cset-swap atomicity (NEW Round 1 2026-05-11 per IM-6)

`SetTXAiqcSwap` (iqc.c:256-269) flips `a->cset = 1 - a->cset`
then memcpy's new coefs into the now-inactive slot.  Thetis
protects this with `EnterCriticalSection(&ch[channel].csDSP)`.
Lyra's Python port via cffi MUST hold an equivalent lock during
the swap — Python's GIL does NOT help here because the WDSP cffi
engine releases the GIL during `xiqc`.

**Bench test:** continuous TX with PS armed; programmatically
trigger `SetTXAiqcSwap` at maximum rate (every 100 ms for 30
seconds).  **Pass:** no audible glitches in the TX I/Q output;
no `cset` mid-block flips visible in iqc output (verifiable by
hooking diagnostic counters to the swap call site and the
xiqc-read site).

#### Gate 7: hw_scale per-radio verification (NEW Round 1 2026-05-11 per IM-6)

`TXA.c:412` sets `hw_scale = 1.0 / 0.4072 ≈ 2.4558` as the WDSP
default — inverse of the PA's full-drive RF magnitude at TX
gain=100%.  This is HL2-specific and may not match for ANAN.

**Bench test:** measure HL2's actual hw_scale on the operator's
specific board (full-drive carrier amplitude at the post-PA
coupler, normalized to expected DDC sample range).  Document the
measured value.  If it differs from WDSP's default by >5%, expose
as a per-radio cal constant in Settings → PS.

**All 7 gates blocking before v0.3 ships** (Round 1 2026-05-11
upgrade from the original 3-gate protocol).

### 9.5 TX panadapter visibility contract (Round 3, 2026-05-11)

**Operator-load-bearing requirement:** the panadapter / waterfall
must never go blank during TX.  The operator transmitting on
20m expects to *see* their carrier, splatter envelope, and (when
PS is active) the post-PA RF leaving the antenna — confirms
frequency, power, modulation quality, and PS predistortion
effectiveness in one view.

This requirement is **not optional**.  Earlier draft of §9 named
the DDC-feed shutdown during HL2 PS+TX (DDC0/DDC1 both pulled to
PS-feedback duty) without spelling out what the operator sees on
the panadapter during that window — which would imply "nothing,"
which is unacceptable UX.  The fix: source-switch the panadapter
input based on MOX + PS state.

**Source matrix (corrected Round 1 2026-05-11 — CR-6 cost,
CR-7 tap point):**

| State | Panadapter source (tap point) | What operator sees |
|---|---|---|
| MOX off | DDC0 (RX1 pan) / DDC1 (RX2 pan) RX-band samples | Live receive (current v0.0.9.x behavior) |
| MOX on, PS off | TX baseband I/Q tapped from WDSP TXA at `sip1` (**post-ALC, pre-iqc** — matches Thetis `xtxa()` line 586) | Self-monitor — the finished modulated waveform after EQ/comp/CESSB/ALC, **before** PureSignal predistortion. | MOX on, PS on (5-DDC ANAN P1) | DDC0+DDC1 twist→source 0 (PS feedback) on RX1 pan; DDC2→source 2 (RX2 stays live!) on RX2 pan | RX1 pan shows post-PA RF; RX2 pan **stays on RX2's band** (ANAN's structural advantage over HL2 — DDC2 is independent) |
| MOX on, PS on (ANAN P2) | Family-specific; populated in v0.4 capabilities struct | Same operator semantic — RX1 pan shows post-PA RF |

- **Waterfall behavior (M-8):** **freeze RX scrollback** at the

  (b) operator-setting: "Replace waterfall during TX" (off by

  On PTT release, RX scrollback resumes from where it was
  frozen, with a faint horizontal line marking the TX gap.
- **Trace badge, not trace color (M-7 polish):** top-right corner
  of each panadapter pane shows a small `TX` badge with a fixed
  accent color (NOT user-customizable, to avoid conflict with
  operator-chosen spectrum trace colors).  Tooltip: "Panadapter
  showing transmitted RF, not band content."
- **Static-overlay grey-out:** all RX-relevant overlays (EiBi
  shortwave broadcasters, NCDXF beacons, Memory bank landmarks)

- **TX-time overlay:** add a new overlay drawing the operator's

- **Cross-backend invariance (M-2 Round 1 2026-05-11):** the

  to prevent the v0.0.9.9.1 EiBi precedent (where one backend's

**RX2 panadapter behavior:** when v0.1 RX2 is enabled, both
panadapters follow the source-switch logic.  On HL2 PS+TX both
panadapters show post-PA feedback content (since DDC0+DDC1 are
both repurposed to feedback duty).  On ANAN P1 5-DDC PS+TX, RX2's
panadapter stays on its band (DDC2 is independent).  Per-RX
badging makes it visually obvious which is which; operator-muted
vs PS-paused use distinct badge iconography per M-4 (§6.5).

**Architectural implication for v0.1 phase 1:** the DDC dispatch
table must be **reconfigurable at runtime on MOX edges**.  Phase
1 doesn't exercise the reconfiguration (no TX code yet), but the
abstraction must be in place so v0.2 TX wiring is additive, not
a refactor.  Specifically: `dispatch_ddc_samples(ddc_idx, samples)`
in `lyra/protocol/stream.py` must consult a state-product
`(mox, ps_armed, rx2_enabled, family)` to decide which consumer
the samples flow to (RX demod chain, PS calibration, panadapter
direct-feed, etc.).  Lyra's existing `freq_source` abstraction
(CLAUDE.md §6.4) is the right shape; phase 1 adds the MOX-edge
state hook.  **Hard-coded `ddc==0 → RX1; ddc==1 → RX2;
ddc==2,3 → discard` in Phase 1 is a smell — the dispatch table
must already be table-driven (consulting `radio.dispatch_state`
state-product) before v0.2 begins.**

**v0.2 TX implementation (corrected Round 1 2026-05-11 — CR-6 cost):**
adds the TX-baseband → panadapter source path.  Tap point is
**post-ALC, pre-iqc** at WDSP TXA's `sip1` siphon (TXA.c:394-403,
`xsiphon` call at TXA.c:586).  **Cost is NOT "a few hundred bytes
per audio block" as earlier draft claimed** — Thetis maintains a
dedicated 16384-sample siphon ring + 16384-pt FFT for the TX
display, working set ~128 KB + ~256 KB FFT scratch + 30-60
FFTs/sec at the display update rate.  Lyra reuses the existing
GPU spectrum widget infrastructure (`lyra/ui/spectrum_gpu.py`)
with a second instance, fed from the TX-side siphon ring buffer.
CPU cost similar to RX panadapter (already proven adequate).

**v0.3 PS implementation:** adds the PS-feedback → panadapter
source path.  When MOX+PS engages on HL2, DDC0/DDC1 sample
dispatch routes to BOTH the PS calibration consumer (calcc)
AND the panadapter direct-feed, so the operator sees what's
being fed to `iqc` while it runs.  On ANAN 5-DDC, the equivalent
routing uses the family-specific PS feedback DDC pair (DDC0+DDC1
twist on ANAN P1) but RX2's pan stays on DDC2's RX2-band content.

## 11. Patterns vs divergences

### 11.1 Adopt verbatim from Thetis

- Channel ID convention (with corrected TX = 4)
- TXFreq derivation pure function (§6.2)
- AAmixer state matrix concept (with Lyra's own indexing scheme)
- Per-channel patchpanel pan curve (`gain1=1, gain2=sin(pan·π)` /
  symmetric)
- iqc 5-state lifecycle (RUN, BEGIN, SWAP, END, DONE)
- PS 8-state command FSM + 3-state AA FSM
- `MuteRX1OnVFOBTX` / `MuteRX2OnVFOATX` as independent operator
  toggles
- Force `VFOATX = true` when SPLIT + RX2 + VFOBTX
- Per-band-per-RX zoom recall pattern
- TUN button label flip ("TUN" ↔ "AUTO")

### 11.2 Deliberate divergences

- Hybrid focus model (single Mode+Filter / DSP+Audio panel + per-
  RX badges + Ctrl+1/Ctrl+2) instead of duplicated controls
- Per-RX display pan instead of global
- Visible AAmixer state badge instead of operator-infers-it
- PS hardware-mod attestation checkbox
- QSplitter for panadapter regions instead of skin percentages
- PS FSM on `QTimer` 10ms instead of `Thread.Sleep(10)` AboveNormal

### 11.3 Do NOT copy

- WinForms Designer .resx-driven layout (no Qt equivalent needed)
- Sub-VFO repurposing the VFO A band label
- Color-only state indicators without text/icon labels (accessibility)
- 8-state FSM with shared mutable globals
- Duplicated control widgets per RX

## 13. v0.0.9.6 Audio Foundation (prerequisite to RX2)

**Slotted in 2026-05-06** after the audio architecture decision
(see `docs/architecture/audio_architecture.md`).  Resolves the
recurring PC Soundcard glitch problem and lays the WDSP-based
audio foundation that RX2 (Phase 2 §5), TX (§8), and PureSignal
(§9) all build on.

### 13.1 What ships in v0.0.9.6

- **WDSP `aamix.c` ported** → `lyra/dsp/mix.py` (replaces the
  planned NumPy stereo mixer — Phase 2 §5 originally called for
  NumPy-only; we're upgrading to the WDSP equivalent because it
  handles the v0.4 ANAN cases that NumPy can't).
- **WDSP `patchpanel.c::SetRXAPanelPan`** ported → into
  `lyra/dsp/mix.py` (RX2 stereo pan curve, prerequisite for
  Phase 2 §5).
- **WDSP `varsamp.c` ported** → `lyra/dsp/varsamp.py` (variable-
  ratio polyphase resampler).  Foundation for PC Sound Card drift
  compensation (this release), TX mic input on ANAN (v0.4), and
  PureSignal feedback rate conversion (v0.3).
- **WDSP `rmatch.c` ported** → `lyra/dsp/rmatch.py` (PI control
  loop wrapping varsamp).  This is the adaptive resampler Thetis
  uses on its ASIO path; absorbs the HL2-vs-soundcard crystal
  drift that has been producing operator-reported audio glitches.
- **Default audio path flip** — for HL2 connections, default
  output is now "HL2 audio jack" (the existing EP2 codec path,
  renamed from "AK4951 mode" since not all HL2 revisions use the
  AK4951 chip specifically).
- **Settings → Audio panel** — operator can override the default
  per-radio.  Hardware capability struct (CLAUDE.md §6.7) drives
  the per-radio default.

### 13.2 What this changes about the v0.1 phases

- **Phase 0 §3:** unchanged — multi-channel scaffolding still
  applies.  v0.0.9.6 ships first; Phase 0 starts after.
- **Phase 2 §5:** `mix.py`'s WDSP `aamix` port replaces the
  planned NumPy implementation.  The pan curve is already there
  (came in with v0.0.9.6).  Phase 2's actual work shrinks to
  "wire the second channel into mix.py" — most of the audio
  routing is done.
- **Phase 3-4 §6-7:** unchanged — UI integration and polish are
  orthogonal.

### 13.3 What this changes about v0.2 / v0.3

- **v0.2 TX (§8):** unchanged for HL2 (mic input via EP6, single
  crystal, no drift).  PC mic input becomes opt-in once
  `rmatch.py` is in place — operators on hardware without an
  HL2 mic jack get a working path.
- **v0.3 PureSignal (§9):** unchanged for HL2 (PS feedback at
  `rx1_rate`, single crystal).  The varsamp foundation makes
  ANAN PS feasible later (`rx1_rate` ↔ `ps_rate=192k`
  conversion).

### 13.4 Why this is in v0.0.9.x and not v0.1.0-pre1

The infrastructure ports could have lived in Phase 0 of v0.1.  We
slotted them into v0.0.9.6 instead because:

1. **Brent's PC Soundcard glitches need a fix sooner**, not after
   the multi-channel refactor.
2. **Cleaner release boundaries.** v0.0.9.6 is "audio
   infrastructure"; v0.1.0-pre1 is "multi-channel scaffolding";
   v0.1.0-pre2 is "RX2 protocol enablement"; etc.  Each release
   tells one story.
3. **Risk isolation.** Audio infrastructure has its own bench
   tests and operator A/B; bugs surface in a release where they
   can be addressed without blocking RX2 work.

### 13.5 Operator capability self-attestation

Default audio path per hardware class:

| Radio class | Default | Rationale |
|---|---|---|
| HL2 (with onboard codec — most modern revs) | HL2 audio jack | Single crystal, no drift |
| HL2 (without codec — early revs) | PC Sound Card with rmatch | Codec unavailable |
| HL2+ | HL2 audio jack | AK4951 codec |
| ANAN G2 / 7000DLE / 8000 (v0.4) | PC Sound Card with rmatch | No onboard codec |

Operator can override in Settings → Audio.  For the early-rev
HL2 case, the protocol module needs to detect "no codec" — most
likely via I2C readback (CLAUDE.md §3.8) OR operator self-
attestation checkbox per the same pattern as the PureSignal
hardware-mod attestation (§6.5).  Detection method TBD; for
v0.0.9.6 we ship the path with operator-self-select default,
add automatic detection in a follow-up if practical.
