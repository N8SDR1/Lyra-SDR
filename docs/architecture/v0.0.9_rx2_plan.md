# Lyra v0.0.9 — RX2 implementation plan

Builds on the Thetis architecture studies at `rx2_research_notes.md`
and `hl2_puresignal_audio_research.md`.  Target: ship dual-receiver
operation with SPLIT TX preview, stereo split audio (RX1 left / RX2
right), and forward-compat hooks so v0.2 PureSignal is a flag-flip,
not a refactor.

This is a **plan** doc.  Decisions that were marked as open in the
first revision are now resolved (per operator input).  The remaining
open items are technical knobs the implementation will figure out.

---

## 1. Scope of v0.0.9

### In scope

- Second receiver (RX2) on HL2's DDC1 — independent VFO B, mode,
  filter, AGC, NR, ANF, NB, squelch.
- **`nddc = 4` protocol topology** — HL2 advertises 4 DDCs on the wire
  even though only 2 are physical.  DDC0=RX1, DDC1=RX2, DDC2/DDC3
  reserved for v0.2 PureSignal feedback.
- SPLIT toggle: TX-source = VFO B while RX continues on VFO A (TX
  itself is v0.1; v0.0.9 wires the SPLIT *concept* and the visual
  TX cursor on VFO B).
- VFO A↔B operations: Copy A→B, Copy B→A, Swap A↔B.
- VFO B lock toggle (prevent accidental tuning while DX-pile-listening).
- **Stereo split audio** — RX1 hard-left, RX2 hard-right via the EP2
  LR bytes through HL2's AK4951 codec.  This is the established right
  approach for dual-RX on HL2.
- L/R swap option in Settings to compensate for HL2 firmware-rev
  variation (`swap_audio_channels` mirror).
- Per-RX pan slider (0=hard-L, 0.5=center, 1=hard-R) with stereo-split
  defaults applied automatically when RX2 turns on.
- RX2 panadapter / waterfall — split central widget vertically; top
  RX1, bottom RX2.
- Persistent state: RX2 enabled/disabled, last RX2 freq+mode+filter,
  per-RX pan, L/R swap, lock state — all in QSettings.
- **Forward-compat hooks for v0.2 PureSignal** (all inert in v0.0.9):
  - DDC frequency-source abstraction per DDC.  In v0.0.9: DDC0=VFOA,
    DDC1=VFOB, DDC2=TX, DDC3=TX.  In v0.2: PS toggles DDC2/DDC3 from
    "static TX freq" to "PS feedback" mode.
  - `puresignal_run` flag plumbed through the C&C writer (sets bit 6
    in frames 11 C2 and 16 C2).  Default false; will toggle in v0.2.
  - Full-duplex bit (frame 0 C4 bit 2) confirmed-set unconditionally
    (it already is, this is just documenting the requirement).

### Out of scope (defer to later releases)

- TX path implementation (v0.1 minimum — no key-up in v0.0.9).
- PureSignal predistortion math + calibration loop (v0.2).
- Full-duplex monitoring during TX (v0.2).
- Sub-receiver inside RX1's DDC (Thetis's `chkEnableMultiRX`) — skip
  unless a strong use case emerges.
- Diversity reception (frame 0 C4 bit 7) — ANAN feature, no clear HL2
  use.
- AK4951 register-level tweaks — gateware owns the codec; nothing to
  do host-side.

### Success criteria

1. Operator enables RX2 from the menu / a button on the Tuning panel.
2. RX1 and RX2 tune independently to any HF freq.
3. Stereo-split audio: RX1 in left ear, RX2 in right ear, both
   simultaneously, with no clicks or dropouts.
4. Each receiver has independent mode, filter, AGC, NR, ANF, NB, SQ.
5. Operator toggles SPLIT, sees VFO B labeled as the TX source
   (cursor on the panadapter, status indicator in the toolbar).
6. Copy A→B, Copy B→A, Swap A↔B all work and update both DSP and
   protocol layers.
7. Settings round-trip cleanly across launch.
8. CPU usage at 192k/192k stays under 25% on a typical tester machine.
9. No regression in v0.0.7 single-RX operation when RX2 is disabled.
10. v0.2 PureSignal can be added by setting `puresignal_run=True` and
    rerouting DDC2/DDC3 frequency-source (no protocol-layer rewrite).

## 2. Architecture

### 2.1 Code organization changes

The v0.0.7 codebase has implicit "one channel" assumptions.  Refactor
first, then add RX2:

- `lyra/radio.py` → introduce a list/dict of channels (`self.channels
  = {0: Channel(...)}`); promote single-channel methods to per-channel
  variants.  Old `self.freq_hz`, `self.mode`, etc. become facades over
  channel 0.
- `lyra/dsp/channel.py` → audit for module-level state (should already
  be instance-clean; class-based).
- `lyra/protocol/stream.py` → add per-DDC frequency state, bump nddc
  from 1 to 4 when RX2 is enabled, EP6 parser for the nddc=4 layout
  (19-sample slots), `puresignal_run` plumbing (inert).
- `lyra/ui/panels.py` → `TuningPanel` already has the placeholder; the
  Mode/Filter/DSP+Audio panels need a "focused channel" concept.

### 2.2 Channel identity convention

Mirror Thetis's `(transceiver, subrx)` tuple but flat int IDs for now:

- **Channel 0 = RX1** (always present)
- **Channel 1 = RX2** (created on enable, persists thereafter; toggle
  `is_active`)
- Reserve 2, 3 for sub-receivers in v0.2+ if needed.

QSettings keys: `rx/0/freq_hz`, `rx/0/mode`, ..., `rx/1/freq_hz`, etc.

### 2.3 DDC topology (corrected from first plan rev)

HL2 always advertises **`nddc = 4`** on the wire.  Lyra's protocol
layer maintains a 4-element DDC freq array regardless of RX2 enable
state:

```
ddc[0].freq_source = "VFOA"   # RX1 — always VFOA
ddc[1].freq_source = "VFOB"   # RX2 — always VFOB (even when RX2 dock disabled, harmless)
ddc[2].freq_source = "TX"     # Reserved for PS feedback in v0.2; static TX freq for now
ddc[3].freq_source = "TX"     # Reserved for PS feedback in v0.2
```

When RX2 is "disabled" from the operator's perspective, we simply
don't *route* DDC1's samples to a Channel — but the wire still
streams them.  Consequence: zero protocol-level cost to hide/show
RX2; the dock visibility and DSP pipeline are independent of nddc.

EP6 parser per the research: with `nddc=4`, `spr = 504/(6*4 + 2) =
504/26 = 19` samples per DDC per USB frame.  Each sample is `[I0(3)
Q0(3) I1(3) Q1(3) I2(3) Q2(3) I3(3) Q3(3) mic(2)]` = 26 bytes.  Two
USB frames per UDP datagram.

### 2.4 DSP threading model

Keep Thetis's pattern: **one network reader thread, all DSP runs on
the worker, each sample callback dispatches to all active channels in
sequence.**

Per-channel DSP at 192 kHz is well within the per-frame budget; two
parallel chains in one thread fits comfortably.  If profiling later
shows we're overrunning, move to a thread-pool model.  Don't
pre-optimize.

### 2.5 Audio routing — stereo split (decided)

Lyra extends its existing AK4951 path from mono RX1 to stereo
RX1+RX2.  The data-flow change:

- Today: RX1 audio → mono → EP2 LR bytes (L=audio, R=audio, mono dup).
- v0.0.9: RX1 audio with `pan=0` → goes to L only.  RX2 audio with
  `pan=1` → goes to R only.  Mixed → EP2 LR bytes (L=RX1, R=RX2).

When RX2 is disabled: pan stays 0/1 but RX2 contributes silence; L=RX1
audio, R=silence.  Operator can still pan RX1 to center (`pan=0.5`)
if they prefer mono-both-ears RX1.

When RX2 is enabled: auto-apply RX1.pan=0, RX2.pan=1 unless operator
has overridden in Settings.

**No host-side sounddevice path** for v0.0.9 — the AK4951 path is the
canonical HL2 audio route and what we already have working.  A
sounddevice option can come in v0.2 if testers want it.

### 2.6 L/R swap option

Per Thetis's `swap_audio_channels` (network.h:110, networkproto1.c:1231):
some HL2 firmware revs swap L and R unexpectedly.  Add a toggle in
Settings → Audio → "Swap L/R audio (HL2 firmware compensation)".
Default OFF; operator flips if they hear RX1 in the right ear instead
of the left.

### 2.7 UI integration — hybrid model (decided)

- Each RX's freq display + panadapter region shows status badges
  (mode, filter, AGC) at a glance — read-only.
- A single "MODE + FILTER" panel and a single "DSP + AUDIO" panel sit
  in their existing layout positions, but operate on the **focused RX**.
- A small RX1/RX2 toggle on the freq-display row (or click anywhere on
  a freq display to focus that RX) switches focus.
- Hotkey: Ctrl+1 / Ctrl+2 to focus RX1 / RX2.
- Focus indicator: colored border around the focused freq display +
  matching tinge on the control panels' headers.

This preserves Lyra's layout density at 1080p; each control panel
stays its current size.

### 2.8 Panadapter / waterfall layout

When RX2 is on, central widget splits vertically:

- Top half: RX1 panadapter + waterfall (current layout).
- Bottom half: RX2 panadapter + waterfall.

Each half:

- VFO cursor for that RX's freq.
- Click-to-tune within that RX's bandwidth.
- Independent zoom / span (RX2 might be at a different sample rate).
- TX-frequency cursor — appears on whichever RX shows the TX VFO
  (per SPLIT).

When RX2 is off, RX1 expands to full central widget.  Splitter handle
between halves is draggable.

### 2.9 Protocol-layer changes

Per the research, EP6 parsing changes substantially with nddc=4.
Plan:

- `stream.py` keeps an `n_ddc` attribute (currently 1, becomes 4).
- Frame 0 writer sets bits 6:3 of C4 to `(n_ddc - 1) << 3 = 0x18`.
- Add a "DDC frequency source" abstraction: each DDC has a
  `freq_source` of type `Enum(VFOA, VFOB, TX, CUSTOM)`.  In v0.0.9:
  DDC0=VFOA, DDC1=VFOB, DDC2=TX, DDC3=TX.  In v0.2 with PS engaged:
  DDC2 and DDC3 source remain TX but the gateware reinterprets them.
- `puresignal_run` flag plumbed: bit 6 of C2 in frames 11 (`0x14`) and
  16 (`0x24`).  Default false in v0.0.9.
- EP6 parser dispatches per-DDC into a `samples_received(ddc_idx, iq)`
  callback, which the Radio routes to the appropriate Channel.  DDC2/
  DDC3 samples are quietly dropped in v0.0.9 (PS not engaged).
- HL2-specific I2C I/O continues unchanged — orthogonal to DDC count.

### 2.10 SPLIT — implementation surface

For v0.0.9 (no TX yet), SPLIT is mostly a UX concept:

- A `chkSplit` toggle on the Tuning panel (or main toolbar).
- When SPLIT is on:
  - TX-cursor on the RX2 panadapter (instead of RX1).
  - Status bar text: "TX → VFO B".
  - Mode + Filter panel's TX-relevant fields (when TX lands in v0.1)
    will follow VFO B.  In v0.0.9 we just plumb the state.
- VFO B lock: separate toggle.  When locked, RX2 freq display shows a
  🔒 icon and rejects clicks/wheel events.

Reified state in `Radio`:

```
@property
def tx_vfo_source(self):  # returns "A" or "B"
    return "B" if self._split_on else "A"
```

Actual TX behavior is v0.1.

### 2.11 Pile-up workflow visualization

Once RX2 + SPLIT are on, the panadapter shows:

- **Top (RX1)**: VFO A cursor (cyan) on the DX op's freq.  Operator
  hears this in the **left ear**.
- **Bottom (RX2)**: VFO B cursor (cyan) on the up-pile freq.  TX-
  cursor (orange/red) overlays on the same freq because TX = VFO B.
  Operator hears the chorus in the **right ear**.

That's the operator workflow: hear DX op's pace in the left ear, hear
the pile-up timing in the right ear, click-and-drag VFO B to land in
a hole, hit TX (when v0.1 ships).

## 3. Implementation phases

### Phase 0 — refactor for multi-channel (no behavior change)

- Convert `Radio` single-channel attrs into a dict keyed by channel ID;
  add facades for backward compat.
- Convert protocol layer to track per-DDC frequency state with the
  `freq_source` enum (only DDC0 used yet; nddc still 1).
- Add `puresignal_run` attribute to protocol layer, default False.
  C&C writer reads it and sets the bit; in v0.0.9 it's always inert.
- Convert UI to read/write through the facades.
- Audit for any lurking single-channel assumptions.
- Smoke test: v0.0.7 behavior fully intact.

Estimated complexity: medium.  Riskiest phase due to breadth; do it
carefully with focused commits.  Could ship as **v0.0.7.1** if it
takes longer than expected.

### Phase 1 — protocol RX2 enablement

- Bump `n_ddc = 4` in frame 0 when RX2 is enabled.
- Wire DDC1 freq into frame 3, DDC2/DDC3 freq into frames 5/6 (both
  static-TX).
- Rewrite EP6 parser for the 26-byte-slot, 19-samples-per-USB-frame
  layout.  DDC0 → RX1 stream, DDC1 → RX2 stream, DDC2/DDC3 → drop.
- Verify on real HL2+: enable RX2, look at received packets in
  Wireshark, confirm DDC0 + DDC1 I/Q reach the parser.
- DSP-side: feed DDC1 samples into the second `Channel` instance.
- Audio: just dump RX2 audio for now (don't worry about routing yet).
- **Empirical test for open question #1**: with PS off, key the radio
  briefly (dummy load) and look at DDC2/DDC3 samples — confirm they
  are at TX freq even with PS inactive.

Estimated complexity: high.  EP6 parser rewrite is exacting — test
with a known-frequency tone on each DDC to catch I/Q swapping or
endianness bugs.

### Phase 2 — audio routing (stereo split)

- Update EP2 audio packer to interleave RX1 and RX2 audio per the LR
  encoding (16-bit BE per channel).
- Add per-RX `pan` parameter (default 0.5 for RX1-only, auto-applied
  to {0, 1} when RX2 enables).
- Add Settings → Audio → "RX1 pan", "RX2 pan", "Swap L/R audio".
- Default behavior: enabling RX2 auto-applies stereo split (RX1=0,
  RX2=1) without operator action.

Estimated complexity: medium.  HL2 audio routing is a known-quirky
area; plan half a day for AK4951 sample-interleave debugging.

### Phase 3 — UI integration

- Promote the dimmed RX2 freq display in `TuningPanel` to live editing.
- Add Ctrl+1 / Ctrl+2 hotkeys for focus.
- Add focus-tinting on the active RX's freq display + matching control
  panel highlight.
- Add A→B, B→A, Swap, RX2-on/off, Split-on/off, VFO-B-lock buttons.
- Status bar: "RX1 14.250 USB | RX2 14.255 USB | TX → VFO A".

Estimated complexity: medium.  Mostly button wiring + state plumbing.

### Phase 4 — split panadapter

- Add a vertical splitter to `lyra/ui/spectrum.py` so the central
  widget can show two stacked panadapters when RX2 is on.
- Each panadapter has its own zoom, span, peak markers, click-to-tune.
- TX-cursor renders on whichever RX is showing the TX VFO.

Estimated complexity: medium-high.  Spectrum widget already complex
(QPainter + GPU paths); adding a second instance + the central
splitter touches both rendering and event-handling code.

### Phase 5 — polish + persistence

- QSettings round-trip for all RX2-related state (incl. pan + L/R swap).
- Help guide updates: `docs/help/tuning.md`, new `docs/help/rx2.md`,
  `docs/help/split.md`.
- About dialog mentions RX2 capability.
- Smoke test: enable / disable RX2 across launches, sleep / wake, etc.
- Regression test: v0.0.7 single-RX flow still pristine.
- Document the `puresignal_run` plumbing as v0.2-deferred; ensure
  flipping it is a single Settings checkbox away (stub the UI).

Estimated complexity: low-medium.  Mostly QA + docs.

## 4. Risk register

| Risk | Mitigation |
|---|---|
| HL2 gateware variant doesn't expose 4 DDCs cleanly | Test on N8SDR's HL2+ early in Phase 1; the existing PS-capable gateware is known to expose 4 DDCs. |
| EP6 parser rewrite has subtle endianness / interleave bugs | Test with two different CW tones on DDC0 and DDC1; verify I/Q dump matches expected freqs. |
| AK4951 stereo split breaks RX1 audio | Keep mono-mode default fallback; if anything misbehaves, AK4951 stays in v0.0.7 mode and RX2 can fall back to silenced (UI shows a warning). |
| L/R swap appears post-ship | The Settings toggle is the answer — operator flips it once. |
| UI feels cramped at 1080p with focus-tinting | Test on real 1080p panel before shipping; soften tints if too busy. |
| Tester confusion about which RX is focused | Make focus indicator unmistakable — colored border on the focused freq display, status bar names the focused RX. |
| Regression in single-RX mode (RX2 off) | Smoke test in Phase 0 + Phase 5; treat any single-RX bug as a blocker. |
| Refactor takes longer than expected, blocking other work | Phase 0 lands as a standalone commit chain; if it stretches, ship as **v0.0.7.1** bug-fix release and continue Phase 1+ on the feature branch. |
| `puresignal_run` accidentally gets set to True somewhere | Default OFF + assertion in tests + UI checkbox not exposed in v0.0.9 (only via QSettings power-user edit). |

## 5. v0.2 PureSignal — the future hook

Once v0.0.9 ships with the abstractions in place, v0.2 PureSignal
becomes:

1. **Protocol** (already plumbed): set `puresignal_run = True`.  C&C
   writer flips bit 6 in frames 11 and 16.  Done.
2. **DSP** (new work in v0.2): WDSP-equivalent predistortion math —
   `calcc.c` / `iqc.c` algorithms.  This is the actual heavy lift.
3. **UI** (new work in v0.2): `PSForm`-equivalent dialog.  PS On / Off,
   Single Cal, Two-Tone, Auto Calibrate, attenuation display, peak
   threshold.  Mirror Thetis's UX.
4. **Empirical verification on N8SDR's HL2+** that PS feedback samples
   actually arrive at TX freq during key-down with the existing
   gateware + mod.

The v0.0.9 architecture should NOT need to change to support v0.2 PS.
That's the whole point of the forward-compat hooks.

### v0.2 won't address

- Operator self-attestation that they have the PS hardware mod
  installed — Thetis doesn't auto-detect this.  Lyra will need a
  Settings checkbox: "I have the PureSignal hardware mod installed on
  my HL2."  Default OFF; until checked, the PS controls are disabled
  with an explanatory tooltip.
- Gateware version detection — Thetis doesn't do this either.
  Operator's responsibility.

## 6. Decision checkpoint

The v0.0.9 plan is now self-contained.  Decisions made (all per
operator input):

- ✅ **Audio**: stereo split via AK4951 EP2 LR bytes (RX1 left, RX2
  right, auto-applied on RX2 enable).
- ✅ **UI**: hybrid model (focused RX + per-RX status badges +
  Ctrl+1/Ctrl+2).
- ✅ **Sub-receiver inside one DDC**: skip.
- ✅ **PureSignal posture**: plan-for-it lightly with forward-compat
  hooks in v0.0.9; full implementation in v0.2.
- ✅ **SPLIT TX cursor**: render now in v0.0.9 even before TX is wired.
- ✅ **Release pacing**: rolling pre-releases per phase.

When ready to start Phase 0:

1. Create a worktree / branch (`feature/rx2-phase-0` off
   `feature/threaded-dsp` or `main`).
2. Refactor for multi-channel (no behavior change).
3. Smoke test against v0.0.7 baseline.
4. Commit + tag as a checkpoint.
5. Move to Phase 1.

Alternatively: review the plan one more time and tell me what to
adjust before Phase 0 starts.
