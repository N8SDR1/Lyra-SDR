# Lyra-SDR Audit — 2026-05-01

Branch: `feature/threaded-dsp` · Audit covers ~33,000 lines of Python.

This audit ran a quantitative DSP sweep across every filter path, traced
state through the Channel/Radio facade, walked threading and Qt signal
flow, and audited QSettings + UI consistency. The codebase is in
generally healthy shape — no FIXMEs/HACKs, no bare-except, no
ModuleNotFoundError-style breakage. Findings below are prioritized by
operator-visible impact.

## TL;DR — what got fixed during the audit

Two concrete bugs were fixed inline (committed below the report):

1. **NR2 wasn't reset on mode switch** — `Channel.set_mode()` only
   called `_nr.reset()`. NR2's decision-directed smoothing
   (α=0.98) would blend in stale spectral state from the previous
   mode for ~500 ms after every LSB↔CW↔AM flip. **Fix**: also call
   `self._nr2.reset()`. Verified — all 129 bins return to seed value.

2. **`load_captured_profile()` was non-atomic across NR1/NR2** —
   if NR2's size check raised after NR1 had already loaded, the
   operator was left with NR1 holding the new profile and NR2
   holding the old one. **Fix**: snapshot prior state, roll back
   on failure, re-raise. Verified — both processors empty after
   intentional size-mismatch failure.

## High-priority findings

### H1 — Audio leveler never reset on mode/freq change
**File**: `lyra/radio.py:828` (leveler instantiated)

The audio leveler lives on `Radio` but Channel doesn't own it,
so when `Channel.set_mode()` runs its reset path, the leveler's
envelope follower (`_env_db`) is left at whatever compressed state
it was in for the previous mode. Audible effect: switching from a
loud LSB signal to a weak CW one means the leveler is still
"holding down" — first 100–200 ms of CW comes out unnaturally
quiet until the envelope releases.

**Fix**: either move leveler ownership into Channel (consistent
with NR/APF/ANF placement) or have `Radio.set_mode()` call
`self._leveler.reset()`. Same applies to `set_freq_hz()` and to
audio-route swaps.

### H2 — No app-shutdown cleanup
**File**: `lyra/ui/app.py` (no `aboutToQuit` hook); `lyra/radio.py` (no `close()`)

When the operator force-quits or the app crashes:
- HL2 stream keeps streaming UDP frames at the radio (the radio
  doesn't know we're gone — it'll keep transmitting at our last
  known TX state until C&C times out)
- AK4951 audio sink doesn't get a final `close()`, which on some
  systems leaves the codec in a buzzy state for ~1 sec on next launch
- QSettings has unflushed writes (band memory, last frequency)

**Fix**: add a `Radio.close()` method that calls `stream.stop()`,
`_audio_sink.close()`, `QSettings().sync()`, and wire it via
`QApplication.instance().aboutToQuit.connect(radio.close)` in
`MainWindow.__init__`.

### H3 — QSettings bool reads use string-tuple workaround pattern
**File**: `lyra/ui/app.py` lines 1853, 1856, 1869, 1872, 1877, and ~12 more

Pattern in current code:
```python
s.value("spectrum_auto_scale") in (True, "true", "True", 1, "1")
```

This works because Windows registry returns strings for bools
written via QSettings, but it's:
- Fragile (any new value the registry returns gets ignored)
- Inconsistent with newer code that uses `s.value(key, False, type=bool)`
  (e.g., `app.py:945`, `1085`)
- Each instance is a re-write of the same coercion logic

**Fix**: search-and-replace the 14+ instances to use
`s.value(key, False, type=bool)`. Single line, safer, idiomatic.
Should be done in a single sweep commit so the pattern doesn't
linger and get copy-pasted into new code.

### H4 — Profile naming inconsistency across DSP modules
**Files**: `lyra/dsp/nb.py:111`, `lyra/dsp/anf.py:117`, `lyra/dsp/leveler.py:102`, `lyra/dsp/nr.py` (NR1 profile)

| Module     | Profile names                              |
|------------|--------------------------------------------|
| NB         | off, **light**, **medium**, **aggressive** |
| ANF        | off, **gentle**, **standard**, **aggressive** |
| NR1        | off, **light**, **medium**, **aggressive** |
| Leveler    | off, **light**, **medium**, **latenight**  |

Three different schemes. ANF stands alone with `gentle`/`standard`,
Leveler stands alone with `latenight`. The Leveler-aggressive case
hit me during the audit — I tested with `aggressive` and it
silently fell back to `off` because Leveler's `set_profile()`
silently rejects unknown names (`set_profile()` line 192).

**Fix**: pick one canonical scheme and migrate. Suggested:
`off` / `light` / `medium` / `aggressive`. Migration path: keep
old names as aliases for one release so QSettings persisted
values still load, then drop in the next.

**Bonus**: change `set_profile()` from silent fallback to
log-and-fallback so future typos surface in the console.

## Medium-priority findings

### M1 — `time.sleep(0.030)` in main thread on audio sink swap
**File**: `lyra/radio.py:3819`, also `4604`

Single-thread mode does:
```python
self._audio_sink.close()
self._request_dsp_reset_channel_only()
import time as _time
_time.sleep(0.030)   # <-- main thread blocks here
self._audio_sink = self._make_sink()
```

30 ms is below the perceptual threshold but it's a UI freeze and
relies on empirical timing rather than a synchronization primitive.
Worker mode handles this correctly (deferred to between blocks).

**Fix priority**: low — it works in practice. But if anyone ever
files a "spectrum stutters when I change audio output" bug, this is
the prime suspect.

### M2 — Cap button right-click menu still references "Show all hidden readouts"
The CPU label's right-click menu was removed in commit `f69597a`
because the Settings checkbox owns CPU visibility. But
`_show_readout_menu` still has the "Show all hidden readouts"
entry — it now only operates on HL2 telemetry, since GPU was
removed entirely and CPU's right-click is gone. The label is
misleading because there's only ever one thing it can unhide.

**Fix**: rename to "Show HL2 telemetry readout" (singular and
specific) or simplify the menu since only one readout has the
right-click affordance now.

### M3 — Status message timeouts vary 3000–6000 ms with no pattern
**Files**: `lyra/radio.py`, `lyra/ui/panels.py` (10+ sites)

Some emits use 3000 ms, some 4000, some 5000, some 6000, some
omit the timeout entirely (defaults to 5000). Operators perceive
some flashes as "too short" and others as "lingering."

**Fix**: standardize. Suggested:
- 3000 ms for routine info ("Saved profile X")
- 4000 ms for important info ("Smart-guard flagged capture as suspect")
- 6000 ms for errors ("Failed to load profile: ...")

### M4 — Dead QSettings keys
**Files**: `lyra/radio.py:397` (`dsp/cw_pitch_hz` written, never read)

Audit found `dsp/cw_pitch_hz` is written by Radio but no read
path uses it; the runtime value comes from Radio's instance var
which has its own default. Either delete the write or wire up
the read in `autoload_*` flow.

`noise/smart_guard_enabled` flagged as similar but I didn't
verify — worth a one-pass check.

### M5 — Settings dialog tab labels say "Network / TCI" with spaces
**File**: `lyra/ui/settings_dialog.py:3433`

Minor: the tab is labelled `"Network / TCI"` (with spaces around
the slash). Other tabs are single words. Doesn't matter functionally
but looks unintentional.

## Low-priority findings

### L1 — NR1/NR2 `_in_buf` phase mismatch when toggling mid-capture
**File**: `lyra/dsp/channel.py` (the `feed_capture` side-channel
introduced in commit `f5d6f03`)

When NR2 is active, NR1's `_in_buf` is fed via `feed_capture()`
to keep the capture accumulator advancing. When NR1 is active,
`_in_buf` is fed via `_nr.process()`. The two paths advance the
buffer the same way under the hood, but if the operator flips
NR1↔NR2 *during* a capture, there's a one-frame phase
discontinuity in the STFT alignment. Audible? Probably not — at
256 FFT / 128 hop / 12 kHz that's ~10 ms of misalignment. But
worth knowing if anyone reports a click on NR-mode flip.

### L2 — Discovery thread emits Qt signals via `Qt.AutoConnection`
**File**: `lyra/ui/discover_probe.py:217`

The `_DiscoveryWorker(QThread)` emits `finished_ok`/`finished_error`
signals. With `Qt.AutoConnection` (default), signals from a QThread
to a main-thread slot auto-marshal correctly. But it'd be more
robust to be explicit (`Qt.QueuedConnection`) so future refactors
that move the receiver don't silently break the pattern.

### L3 — `bare except` count: 73 across 20 files
None are truly bare (`except:`) — they're all `except Exception:`
with mostly-justifiable logic. Worth a future pass to convert
the print-and-swallow ones (mostly in noise_profile_store.py and
panels.py file I/O paths) to specific exception types.

### L4 — `lyra/__init__.py` is 58 lines but several other `__init__.py` files are 0 bytes
Not a bug, but the empty ones (`lyra/protocol/__init__.py`,
`lyra/ui/__init__.py`) make `from lyra.protocol import discovery`
require explicit re-exports. Consider adding minimal exports for
discoverability.

## Architecture observations (not bugs, just notes)

- **Radio is ~5000 lines** — the central facade has accreted a lot.
  Worth considering breaking out:
  - Telemetry/HL2-status code → `radio_telemetry.py`
  - Spectrum-FFT timer + slot → `radio_spectrum.py`
  - Worker-thread sink swap dance → `radio_threading.py`

- **panels.py is ~4100 lines** — same situation. Each panel
  (DspPanel, TuningPanel, etc.) could be its own file. The
  recent `_prompt_profile_name` helper at module-top is a good
  template for shared utilities going into `panels_common.py`.

- **`worker.py` is 700 lines for an opt-in beta path** that most
  operators don't use. Until it ships as the default, consider
  marking it more clearly as experimental in the Settings tooltip.

- **Profile system is healthy**: smart-guard variance check, atomic
  JSON writes, `fft_size` compatibility check, age coloring on the
  badge. Good design; the only friction is the recently-added
  24-char name cap, which lands well.

## What I'm leaving alone

- The DSP math itself — Ephraim-Malah implementation, NR1's
  spectral subtraction, ANF's leaky LMS, NB's bg tracker. Each is
  textbook-correct and matches verifiable RMS deltas in the sweep.

- The HPSDR P1 protocol layer — not touched, audited only at the
  threading boundary.

- The spectrum/waterfall paint code — runs on Qt's render loop,
  performance-critical, not part of operator-facing complaints.

## Recommended fix order

1. **H1** (Leveler reset) — small, audible, one line
2. **H2** (App shutdown hook) — small, prevents the "buzzy AK4951 on
   relaunch" symptom
3. **H4** (profile naming) — medium-effort migration, big UX win
4. **H3** (QSettings type=bool) — search/replace, no behavior change
5. **M1–M5** as time permits
6. **Refactor/split radio.py + panels.py** — nice-to-have, helps
   future feature velocity
